Executive Brief — Bottom Line Up Front

The problem. Agentforce™ flex credits meter every AI action. Without governance, a single runaway subagent, a prompt regression, or a viral use case can burn a quarter’s budget in days — with no native ceiling, no auto-throttle, and no executive-friendly dashboard.

The gateway. A policy layer that sits in front of your agent runtime, tracks estimated credit consumption in real time, and progressively dials back AI capability as the budget tightens — before hitting the ceiling. When the budget is healthy, agents operate at full power. When it tightens, they step down model tier and cache responses. At zero, new LLM calls are blocked entirely and complex requests route to a live human.

The decision. Two delivery paths. Baseline (pure Salesforce®) ships in 6–8 weeks for ~42 person-days and covers 80% of governance needs. Enterprise (hybrid with an off-platform service) adds 4–6 weeks for real-time latency, auto-calibration, multi-org rollup, and ML forecasting. Most organizations start with Baseline; Enterprise is the upgrade once volume or multi-org operation demands it.

The investment. One-time build: ~42 person-days (Baseline) or ~75 person-days (Enterprise). Ongoing: ~2–4 hours/week of admin time, plus $200–$800/month cloud infrastructure (Enterprise only). No additional Salesforce® licenses required beyond Agentforce™ flex credits you already own.

Salesforce® roadmap context. Salesforce®’s Consumption Tags framework (standard + custom tags, three associated permissions) is GA today and provides attribution for consumption. A Flex-Credits-specific tagging enhancement is publicly targeted for June 2026. Neither shipped nor announced is any consumption-driven hard-stop or programmatic enforcement API. The gateway provides enforcement; Salesforce provides attribution. Complementary, not competing — see Salesforce Roadmap.

Gateway Overview — The Five Moving Parts

Stripped of implementation detail, the gateway is five interacting components. Each has a single, easy-to-explain job.

1. Consumption Tracker

Counts estimated credit spend as agents work. The “meter” on the wall.

2. Budget Phase Engine

The “thermostat.” Watches the meter, declares the current phase (Abundant / Conservative / Emergency / Exceeded).

3. Circuit Breaker

The “safety fuse.” Trips on error bursts or hard-zero budget; reroutes to humans or cached answers.

4. Model Tier Router

The “traffic cop.” Routes each request to the cheapest model that can handle it given the current phase.

5. Observability Layer

The “control panel.” Dashboards, alerts, and forecasts for finance, product, and ops.

You can deploy them incrementally. The Consumption Tracker + Budget Phase Engine alone already prevent runaway spend. The rest is progressive upside.

Flex Credit Cost Model — How the Gateway Measures Spend

The gateway keeps its own running count of estimated credit consumption (the “shadow ledger”) and reconciles it against Salesforce®’s authoritative usage data periodically. This matters commercially because:

  • Real-time visibility — you see what’s being spent today, not in next month’s invoice.
  • Safety band — the gateway holds a ~10% margin below the official ceiling so estimation drift never causes a surprise overage.
  • Calibration discipline — a scheduled reconciliation compares estimated vs. actual; if drift grows, the gateway auto-tunes its multipliers (Enterprise) or prompts an admin to do so (Baseline).

Bottom line for finance: The shadow ledger turns an opaque monthly invoice into a live burn-rate meter — same mental model you already use for cloud-compute budgets.

Budget Phase Engine — The Four-Phase Thermostat

As the budget draws down, the gateway shifts through four graduated phases. Each phase adjusts what the AI is allowed to do — starting from full power and progressively favoring cheaper models, cached answers, and human handoff. The result: service never goes dark, cost never runs away.

● Abundant · > 60% left

Full power. Best models, multi-step reasoning, proactive tool use. The gateway gets out of the way.

● Conservative · 20–60% left

Invisible trims. Smaller models, tighter context, no speculative work. Users don’t notice; your P&L does.

● Emergency · < 20% left

Protect the ceiling. Smallest models only, cache-first, human queue pre-warmed. Alerts fire.

● Exceeded · 0%

AI off, humans on. Every new request is either a template answer or an Omni-Channel® handoff. Zero overage.

Policy decision that needs an executive signature: What happens at 0%? Three choices — hard stop, HITL-only, or auto-purchase more credits with a workflow approval. The gateway supports all three; the decision is a business call.

Circuit Breaker — The Safety Fuse

Borrowed from distributed-systems engineering (and from your home electrical panel). If the agent runtime starts producing errors — or if the budget hits zero — the circuit “trips” and all complex traffic diverts to a safe fallback until the underlying condition clears. This closes two business risks at once:

  • Risk #1 — Cascading failures. A misbehaving agent is unlikely to self-correct; every retry costs credits. The breaker stops the bleeding.
  • Risk #2 — Zero-budget cliff. At 0% budget the breaker hard-opens; no new LLM call can be made until credits are replenished or the phase recovers.

End-user experience: When the breaker is open, customers still get helped — either by a cached answer (for simple questions) or by a live human via Omni-Channel® routing. The degradation is graceful, not a dead silence.

Model Tiering — The Biggest Single Cost Lever

Agentforce™ supports multiple model tiers (small / medium / large), with roughly a 3–10× cost difference between the cheapest and most expensive. In practice, most agent steps — classification, routing, short acknowledgements, FAQ lookups — work perfectly on the cheapest tier. Using large models everywhere is the single most common source of cost waste.

The gateway’s Model Tier Router routes each request to the smallest model that can handle it, with the phase engine as a further ceiling (in Emergency, only small models run at all).

Observed cost reduction from tiering alone: 40–60%. This is the benefit most CIOs underestimate when scoping an AI cost program.

Guardrails That Cost Nothing to Enforce

A non-obvious but commercially important detail: the gateway’s decision to spend or not spend a credit is itself free. The Agent Script DSL that drives Agentforce™ allows conditional branches, data lookups, and template responses to execute at zero credit cost — credits are only consumed when an LLM actually runs.

Concretely, this means the gateway can:

  • Check the current phase before the LLM is invoked (free).
  • Return a deterministic template answer when the phase is Exceeded (free).
  • Route to a human via Omni-Channel® when the breaker is open (free).

The takeaway: the gateway never spends credits to decide not to spend credits. Its overhead is essentially zero — it’s pure downside protection.

Enterprise: Off-Platform Gateway Service

A small Node.js microservice running on Google Cloud Run™ (or AWS® / Azure® equivalents). It handles consumption ingestion, budget phase computation, circuit-breaker state, and forecasting — and exports OpenTelemetry to whatever observability backend your SRE team already runs.

The service is designed so that if it goes down, Salesforce® keeps working. The on-org Baseline components remain fully functional and continue to enforce budget policy; the off-platform layer is an enhancer, not a single point of failure.

Enterprise: Integration Layer

Authentication is OAuth JWT Bearer flow between the gateway service and Salesforce® — a server-to-server pattern with certificate-based credentials, no user passwords, and IP allowlisting for egress control. Usage events flow from Salesforce® to the gateway via the Pub/Sub API (gRPC streaming), which does not count against daily API limits.

Testing — How We Know It Works

Three layers of validation, same pattern your quality team already uses for mission-critical Salesforce® deployments:

  • Unit tests — 50–80 automated tests at the code level, ≥95% coverage. Runs on every deployment.
  • Integration tests — 15–25 end-to-end tests that drive all four budget phases, trip the circuit breaker, and validate human-handoff.
  • User acceptance — 5–10 business-scenario UAT walkthroughs before production cutover.

The two UAT tests that most reassure executives: (1) “Agent gracefully degrades in Emergency phase” — users get a working, smaller AI, not an error. (2) “Agent routes to a human in Exceeded phase” — a real Omni-Channel® case is created and a live human receives the handoff. Both must pass before go-live.

The Problem: Unbounded AI Consumption in Enterprise Agentforce™ Deployments

For Business Leaders: Every Agentforce conversation costs flex credits. Without guardrails, a single runaway agent — or a spike in user adoption — can exhaust your quarterly AI budget in days. This whitepaper shows you how to build an intelligent “cost thermostat” that automatically adjusts AI behavior as spending approaches your budget ceiling, ensuring continuous service while preventing overages.

Salesforce® Agentforce™ operates on a flex credit consumption model where each agent conversation, Einstein™ AI interaction, and Data Cloud™ query deducts from a shared organizational credit pool. As enterprises scale their Agentforce deployments across Service Cloud™, Sales Cloud™, and custom agents, consumption becomes unpredictable due to:

  • Variable conversation depth — Simple FAQ lookups cost a fraction of complex multi-turn problem resolution sessions. A single escalation path through multiple sub-agents can consume 10–50× the credits of a standard interaction.
  • Agentic loop amplification — Agents with write-capable tools (record updates, external API calls, approval routing) can enter retry loops that compound consumption exponentially before hitting platform timeouts.
  • Model tier opacity — The Agentforce runtime selects foundation model capacity dynamically. Without explicit governance, expensive large-model invocations may fire for tasks that a smaller model could handle at 1/10th the cost.
  • Shadow agent proliferation — As citizen developers build custom agents via Agent Builder, consumption becomes distributed across dozens of uncoordinated agent instances with no centralized budget view.
  • No native budget enforcement — As of April 2026, Salesforce provides consumption visibility (Digital Wallet, Usage-Based Entitlements, Data Lake Object queries), a shipped tagging framework (Standard + Custom Consumption Tags), and configurable threshold alerts, but no mechanism to proactively throttle, cap, or redirect agent behavior when spending approaches a limit. Agents continue operating — and consuming credits — past any alert threshold. Overage charges accrue automatically per the Order Form. The next publicly stated milestone is a Flex-Credits-specific tagging enhancement targeted for June 2026 (see Salesforce Roadmap); it does not change the enforcement gap.
  • The only platform-native enforcement option is manual agent deactivation in Setup — an all-or-nothing global kill with no consumption trigger, no graduated response, and no HITL fallback. It depends on a human noticing in time, and when invoked it shuts off the feature for every user simultaneously. That is the blunt instrument the gateway is positioned to replace with phase-based step-down, job-type prioritization, and circuit-breaker routing to a human agent.

The TenantUsageEntitlement object and Setup > Usage-Based Entitlements page provide post-hoc consumption data (daily refresh), but expose no trigger-compatible event or API that an Apex governor could subscribe to in real-time. Digital Wallet offers near-real-time visibility and a Consumption Threshold Alerts Flow that can fire notifications, but it has no enforcement mechanism — it cannot block, throttle, or redirect agent actions. The DigitalWallet standard object (Commerce Cloud, API v48.0+) is unrelated; it tracks customer-facing payment wallets, not the org’s flex credit ledger. This enforcement gap is what the Cost Management Gateway addresses.

Flex Credits: How Consumption Works

For Business Leaders: Salesforce® introduced Flex Credits in May 2025 as a replacement for the original $2-per-conversation pricing model. Instead of paying a flat fee per chat session, your organization now pays per action — a specific function an AI agent executes, such as updating a record, summarizing a case, or executing a prompt. Simple interactions that require only 1–2 actions cost pennies; complex multi-step resolutions cost proportionally more. This aligns cost to value: you pay for what the AI actually does, not how long the user chats.

What Is an Action (Tool)?

An action — also referred to as a tool when presented to the LLM reasoning engine — is any discrete function an Agentforce™ agent executes on the platform. This includes Flow invocations, Apex @InvocableMethod calls, Prompt Template executions, record lookups, and any custom or standard action configured in a subagent (formerly topic).

What Is a Token?

For Business Leaders: A token is the unit of text that an AI model reads and writes. Think of it as a syllable-sized chunk of language — the word “customer” is two tokens, “the” is one, and a technical term like “CircuitBreakerService” might be four. When an agent reads a customer’s message, looks up a knowledge article, reasons about the answer, and writes a response, every piece of that text is measured in tokens. More tokens means more AI compute, which means more cost.

Token Approximation: For English text, 1 token ≈ 4 characters ≈ 0.75 words. A 10,000-token budget covers roughly 7,500 words or ~22 pages of standard prose. Both the input (instructions, context, user message) and the output (agent response) count toward the token total. Actions that retrieve large knowledge articles or generate long responses are the primary drivers of token overflow.

Each action processes up to 10,000 tokens (~7,500 words). Actions exceeding this threshold are billed as multiple actions:

Monitoring: Digital Wallet — Complete Capabilities

Salesforce® provides Digital Wallet — a free account management tool offering near real-time consumption data for Flex Credits across active contracts. Access it via App Launcher → Digital Wallet or Setup → Digital Wallet. Requires the View Consumption user permission (assigned via profile or permission set — per Digital Wallet Permissions; verified 2026-04-28). The sibling View License Utilization permission grants access to seat-based product cards, distinct from consumption cards.

What Digital Wallet Tracks

Digital Wallet provides consumption cards for: Agentforce (Sales and Service), Data Cloud (including sandbox), Einstein Requests, Einstein Personalization, Salesforce Messaging, Unified Messaging, and Tableau® Unified Analytics. Each card shows units consumed, remaining capacity, multipliers, and usage trends over configurable timeframes (7, 30, 90 days). A Monthly Account Summary email is sent to billing contacts on the 10th of each month.

Consumption Threshold Alerts

Digital Wallet includes a ready-to-use Consumption Threshold Alerts Flow that fires when consumption on any card reaches a configurable percentage threshold. Default actions include in-app notifications and email alerts, with optional Slack® and desktop notification channels. The threshold alerts operate on the total card balance and cannot distinguish between sandbox and production usage.

Gateway Integration Opportunity: The Consumption Threshold Alerts Flow is customizable. It can be extended to: publish a Platform Event (enabling the Cost Management Gateway to subscribe and react), invoke Apex actions, or make Connect API calls to pause Data Cloud processes. This is the primary hook the gateway uses to synchronize its shadow ledger with actual consumption data.

Critical Limitation for Agentforce: The Connect API pause pattern (changing a process schedule to “No Schedule”) works for Data Cloud processes only — data streams, identity resolution, calculated insights, and data transforms. There is no documented Connect API endpoint to pause, throttle, or deactivate Agentforce agents at runtime. This is the enforcement gap the Cost Management Gateway fills with its Apex-based circuit breaker and model tier routing (see Circuit Breaker and Model Tiering).

Gateway Calibration Strategy by Path:

Baseline (no Data Cloud): The gateway relies on TenantUsageEntitlement (daily SOQL, no extra license) as the calibration source. The Consumption Threshold Alerts Flow is customized to publish an AI_Budget_Phase_Change__e Platform Event when thresholds are hit, giving the gateway near-real-time awareness of consumption spikes in production (the flow cannot run in sandbox — see warning below). An admin manually enters the current credit balance weekly as a backup calibration point.

Enterprise (with Data Cloud): The off-platform gateway queries multiple DLOs: TenantHourlyEntitlementConsumption for continuous hourly burn-rate signal, TenantEnrichedUsageEvent for resource-level attribution and consumption tags, TenantEntitlementTransaction for credit balance and contract-pooling order, and TenantUsageTypeMultiplier as the authoritative multiplier source (replaces hand- maintained values in AI_Cost_Config__mdt). The Threshold Alerts Flow additionally publishes Platform Events for immediate phase-transition triggers via the Pub/Sub API. For per-LLM- call token attribution, the gateway optionally taps the adjacent Einstein Audit GenAIGatewayRequest DMO via Custom Report Type queries.

Both paths: TenantUsageEntitlement serves as the daily reconciliation checkpoint, since DLO data is “pending” and may differ from settled billing.

Sandbox testing: the threshold-alerts hook is unavailable. The gateway must fall back to the DLO query path (or TenantUsageEntitlement in Baseline) for sandbox calibration. Plan for hybrid testing: production uses the full hook chain; sandbox uses query-only.

Sandbox limitation — threshold-alerts flow does NOT run in sandbox. Per xcloud.wallet_get_notified.htm: “You can access the Consumption Threshold Alerts flow in a sandbox, but you won’t be able to run it there. The flow fails because Digital Wallet invocable actions aren’t supported in sandbox environments.” The gateway’s sandbox readiness checklist must therefore exercise the DLO-query calibration path, not the threshold-alerts path. The threshold-alerts hook is verified only in production rollout.

Opportunity Cost — The Hidden Tax of the Wrong Model

Flex credits are only the measured cost. Every time the wrong model is chosen for a step, a set of opportunity costs pile on behind the visible credit charge. These costs rarely show up on the invoice, yet they compound month over month and can dwarf the raw credit spend they sit alongside. The gateway’s Model Tier Router and Budget Phase Engine both exist in large part to drive these opportunity costs toward zero.

Opportunity-Cost FactorWhat Goes Wrong When the Model Is Mis-MatchedTypical Magnitude
Latency — time-to-first-token (TTFT) Large models take 2–5× longer to begin streaming than small ones. For chat channels, every 500ms past ∼1s of TTFT measurably drops conversational quality and increases user abandonment. Using Large for trivial classification/routing adds 2–8 seconds per turn that the user waits for no reason; abandonment rises 5–15%.
Latency — end-to-end response time Large models generate tokens slower (lower tok/s) and often generate more tokens. Both effects compound. Simple FAQ answers can take 8–20 seconds on Large when Small delivers in 1–3. Perceived as a broken agent.
Context window waste Stuffing a small-model’s job into a Large model’s 128K+ context means paying for attention over content that doesn’t improve the answer. Context-filled Large calls can burn 10–30× the credits of a focused Small call for identical output quality.
Throughput / concurrency ceiling Every model tier has its own provisioned-throughput rate limit. Large-tier limits are lower, so heavy Large usage at peak hits throttling before flex credits are exhausted. Throttling manifests as HTTP 429s and queue backups during peaks — visible to end users and on-call, not to finance.
Human escalation friction Slow or poorly-matched AI increases the rate of customer fallback to humans before the circuit breaker activates — burning AI credits and HITL capacity for the same interaction. Poor model fit can double the effective cost of every conversation by consuming both channels.
Quality regressions Counterintuitively, larger isn’t always better. Large models over-think simple routing decisions, hallucinate process steps small models correctly stay silent on, and generate verbose answers for yes/no questions. Measured accuracy drops on constrained tasks (intent classification, slot filling, tool-argument generation) when forced onto a too-large model.
Retry amplification When a model fails a structured-output or tool-call format, the agent retries — each retry a full billable action. A mis-matched model fails these more often. Bad model fit produces 1.5–3× the retry rate of a well-matched tier.
Energy & ESG exposure Large-model inference consumes proportionally more compute energy. For ESG-reporting organizations, unnecessary Large usage adds to disclosed Scope 3 emissions. Typically 2–5× per-call kWh for Large vs. Small; annualized at scale this becomes material.
Vendor lock-in / portability Building agents that only function correctly on a single vendor’s top-tier model makes the agent expensive to port or re-benchmark when pricing changes or a better provider emerges. Re-platforming a monolithic-Large agent can take months; a tiered agent is portable in weeks.

Decision heuristic for model selection: match the model to the task’s hardest step, not to the agent as a whole. The rule of thumb that produces the best outcomes in practice — “small for classification, medium for generation, large only for reasoning or long-context synthesis” — drives both direct credit cost and opportunity cost toward the floor at once. See the Model Tiering Strategy section for the explicit tier map and routing rules.

Budget Phase Engine — Graduated Enforcement

For Business Leaders: The Budget Phase Engine is the “thermostat” at the heart of the gateway. It continuously monitors your spending against your budget ceiling and automatically shifts the organization through four graduated phases. Each phase adjusts what the AI agents are allowed to do, progressively reducing cost while maintaining service continuity. The key insight: never allow unchecked overage charges — instead, gracefully dial capability down as spending approaches the ceiling.

Reading the Phase Diagram — Plain-English Walkthrough

The four boxes above are the budget phases. Arrows show the transitions that move the gateway from one phase to the next — downward as budget is consumed, upward as it’s restored. Inside each phase the gateway enforces a specific behavioral posture (model tier ceiling, context budget, tool policy). Those postures use camelCase identifiers in the implementation; their plain-English meaning is spelled out below.

● Abundant — “Go fast”

Budget is healthy (> 60% remaining). The gateway gets out of the way — agents use the best available models, assemble full conversation context, and may proactively invoke tools to be helpful.

  • FullAI — any model tier allowed (including the most expensive)
  • MultiHop — agent may chain multiple reasoning steps and tool calls
  • SpeculativeTools — agent may call tools the user didn’t explicitly ask for, if it anticipates a need
● Conservative — “Stay sharp”

Budget is between 20–60%. The gateway starts trimming cost-heavy behaviors without the user noticing — smaller models, tighter context, no speculative work.

  • MidTier — medium models preferred; small models used for classification/routing
  • SingleHop — simplified context only; no multi-turn chaining by default
  • NoSpeculative — agent only calls tools when explicitly asked (reactive, not proactive)
● Emergency — “Protect the ceiling”

Budget is below 20%. The gateway aggressively preserves remaining credits — small models only, prefer cached answers, and keep a human agent on standby.

  • SmallOnly — only the cheapest model tier; medium and large calls are blocked
  • CacheFirst — if a similar question was already answered, return that instead of a new LLM call
  • HITLReady — Omni-Channel® queue pre-warmed so humans can take over instantly if needed
● Exceeded — “AI off, humans on”

Budget has hit 0%. The gateway refuses all new LLM invocations — users either get a deterministic template answer or are routed to a human agent. No overage charges.

  • NoNewAI — every incoming request is short-circuited before it reaches the model
  • DeterministicOnly — simple requests get a pre-written template response
  • HITLRoute — anything complex is routed to a live human via Omni-Channel®

Implementation: The phase is computed as: phase = ratio > 0.6 ? 'ABUNDANT' : ratio > 0.2 ? 'CONSERVATIVE' : ratio > 0 ? 'EMERGENCY' : 'EXCEEDED' where ratio = 1 - (consumed_credits / budget_ceiling). Policy Decision (requires Executive Sponsor): The most critical business decision is what happens in the Exceeded phase. Options: (1) hard stop — all agent channels display a “service temporarily limited” message; (2) HITL-only — all requests route to human agents; (3) emergency credit purchase — auto-request additional credits with approval workflow. This is a business policy, not a technical decision. The gateway supports all three.

Circuit Breaker Pattern

For Business Leaders: A circuit breaker is like a safety fuse in your electrical panel. If the AI agent encounters too many errors in a row — or if spending suddenly spikes — the circuit “trips” and temporarily redirects all traffic to a safe fallback (human agents or cached responses). After a cooldown period, it cautiously tests whether the problem is resolved before fully restoring AI service.

stateDiagram-v2
  direction LR

  [*] --> Closed : Service startup
  Closed --> Open : 3 failures in 60s OR phase = Exceeded
  Open --> HalfOpen : 5-minute cooldown elapses
  HalfOpen --> Closed : Probe succeeds AND phase is not Exceeded
  HalfOpen --> Open : Probe fails OR budget still Exceeded

State Reference — What’s Happening in Each State

● CLOSED — healthy

All traffic flows to the Agentforce™ runtime. The breaker tracks a running success count and a rolling failure window so it can detect trouble before it compounds. This is the normal, steady-state operating mode.

● OPEN — tripped, fail-fast

All traffic diverts to the fallback path (HITL or cached response). No new LLM calls are allowed. End users see a graceful “service limited” message; complex requests are automatically routed to a live human agent via Omni-Channel®.

● HALF-OPEN — probing

The cooldown timer has elapsed. Exactly one probe request is allowed through to test whether the upstream problem has cleared — this prevents a thundering-herd recovery where full traffic floods a still-degraded runtime.

Model Tiering & Step-Down Strategy

For Business Leaders: Under Salesforce’s current Flex Credit pricing, all standard Agentforce actions cost 20 credits regardless of which foundation model processes them. The cost savings from model tiering come from three levers: (1) smaller models resolve simple queries in fewer actions (1–2 vs. 5–8), directly reducing credit consumption; (2) smaller models produce fewer token overflows (exceeding 10,000 tokens per action doubles the billing); and (3) in constrained budget phases, the gateway routes to deterministic template responses that consume zero AI credits.

BYOLLM Credit Reduction: Salesforce’s Supported Models documentation states that Bring Your Own LLM (BYOLLM) configurations consume 30% fewer Einstein Requests compared to Salesforce-managed models. Einstein Requests are the legacy billing metric that preceded Flex Credits. Whether this 30% reduction carries forward to the Flex Credit billing model should be verified with your Salesforce Account Executive before factoring it into budget projections. If confirmed, BYOLLM model selection becomes a significant cost optimization lever — potentially reducing per-action credit consumption from 20 to 14 credits. Supported BYOLLM providers: Amazon Bedrock™, Azure® OpenAI, OpenAI, and Vertex AI™ (Google). Note that BYOLLM still routes through the Salesforce Models API and Einstein Trust Layer; you additionally pay your LLM provider’s inference fees.

Trust Layer Composition — What the Gateway Must NOT Log

The Einstein Trust Layer applies to all Agentforce sessions and is not configurable to “off”. The gateway’s logging design must compose with it:

  • Do not log raw prompts. The Trust Layer masks PII before prompts reach external LLMs (Claude / OpenAI / Gemini / Bedrock). The gateway sees the masked stream; log that, not the pre-mask form.
  • Do not log raw responses. Same rationale: the Trust Layer applies data-detection re-identification on responses. Log token counts and trust-layer disposition flags, not response bodies.
  • Zero data retention with LLM providers is a Salesforce® commercial guarantee with the model providers. The gateway’s long-lived storage of any prompt content would invalidate that guarantee for the customer’s org. Stick to metadata about the request: estimated credits, model tier, agent identity, consumption-tag bag, conversation ID, phase at time of execution.
  • FLS / sharing rule compliance is enforced by Trust Layer dynamic grounding. The gateway has no FLS bypass — if a user couldn’t see a record without the gateway, they still can’t see it via gateway-mediated requests.

Agent Script™ Credit Model — The Most Important Cost Lever

For Business Leaders: Not all agent operations cost the same. In fact, most of the “plumbing” — routing decisions, variable checks, state transitions — is completely free. Only actual AI model calls and action executions cost credits. This means a well-designed cost gateway can perform all its budget checks, phase computations, and routing decisions at zero credit cost, only consuming credits when the agent actually does productive work.

The Zero-Cost Budget Gate Pattern (Agent Script)

Critical Design Pattern: The before_reasoning: lifecycle hook executes deterministically (not LLM-directed) at zero credit cost. Combined with the available when @variables.budget_remaining > 0 action guard, this provides an enforceable, deterministic budget gate that the LLM cannot bypass.

This is fundamentally different from relying on agent instructions like “check the budget before executing actions.” Instructions are probabilistic (Layer 3) — the LLM may ignore them under certain reasoning paths. The available when guard is deterministic (Layer 4) — if the condition is false, the action literally does not appear in the agent’s available action list, making it impossible to invoke.

The implementation uses three Agent Script mechanisms working together: (1) a before_reasoning: hook that checks the budget phase before the LLM begins processing — at zero credit cost — and immediately transitions to a deterministic fallback subagent if the budget is Exceeded; (2) an available when @variables.budget_remaining > 0 guard on every paid action, which physically removes the action from the agent’s available list when the budget is exhausted (the LLM cannot invoke what it cannot see); and (3) a free-tier fallback action that is always available in constrained phases, providing template-based responses at zero credit cost.

Key Insight: This is the single most impactful cost optimization pattern available in the Agentforce platform. The budget gate runs before any LLM reasoning, costs zero credits, and uses deterministic guards that the AI cannot override. The full Agent Script implementation pattern is provided in Appendix D.

Enterprise · Implementation Approach #2Hybrid On-Org / Off-Platform Implementation

For Business Leaders: Enterprise extends Baseline by adding an external microservice that handles the heavy lifting — real-time credit monitoring, ML-based forecasting, automated calibration, and cross-org budget management. Crucially, Enterprise is the tier where cost governance stops being an “Agentforce-only” concern and expands to cover the entire organization’s Salesforce®-hosted AI footprint — one budget envelope, one dashboard, one enforcement policy for every AI-driven workflow in every production org. The benefits accrue to both single-org and multi-org architectures: single-org customers gain organization-wide AI-spend control beyond Agentforce™; multi-org customers gain centralized, audit-ready rollup across every region, business unit, and sandbox that shares the same flex-credit wallet.

Scope — What the Enterprise Tier Actually Gates

● Single-Org Deployments

Even with a single production org, Enterprise is the tier at which the gateway’s scope crosses the Agentforce™-only boundary. Once the off-platform service is in place, the same budget-phase, circuit-breaker, and model-tiering policies can govern every other AI feature that meters against the flex-credit wallet — Prompt Builder templates fired from Flows, Einstein™ generative actions, custom LLM invocations from Apex™ and Data Cloud™ activations, Marketing Cloud® send-time AI, and anything else that bills against the same wallet. One policy layer, one budget, one dashboard for every AI line item on the Salesforce® P&L.

Why this matters: customers who start with a single Agentforce™ deployment typically add 2–4 additional AI surfaces within 12 months. Enterprise keeps all of them inside the same governance envelope as they come online — no re-architecture required.

● Multi-Org Deployments

Enterprise is the only tier that supports multi-org cost governance. One off-platform gateway service subscribes to Platform Events from every production org you operate — regional prod orgs (NA / EMEA / APAC), business-unit orgs from acquisitions, dev sandboxes, staging, UAT — and rolls all of them up into a single budget ledger. Enforcement happens at the org level: dev sandboxes might be capped at 5% of the budget, staging at 10%, production at 85% — all tuned centrally, all visible on one executive dashboard.

Why this matters: regulated industries (financial services, healthcare, public sector) routinely require a single source of truth for AI spend across all subsidiaries and regions. Baseline can’t provide that — it governs one org at a time. Enterprise is the only tier that meets the audit requirement.

Why Enterprise — What You Actually Get Back

Baseline is a complete, production-ready cost gateway. Enterprise is not a replacement — it is an amplifier. Everything Baseline does, Enterprise does faster, more accurately, and hands-free. Below is the concrete delta, framed as what changes for the business, the admin, and the end user.

● Accuracy — no more drift

Baseline’s shadow ledger is an estimate that an admin manually calibrates weekly. Enterprise reconciles estimated vs. actual consumption every night, automatically, via direct API scraping of Usage-Based Entitlements. Drift stays under ±5% continuously instead of oscillating to ±25% between manual calibrations.

● Latency — < 10ms budget checks

Baseline budget checks are Apex + SOQL: ~100–500ms per check, and they count against governor limits. Enterprise answers “what phase are we in?” from a Redis™ cache in under 10ms, with no governor-limit exposure. At high volume this is the difference between agents that feel snappy and agents that feel sluggish.

● Forecasting — days-until-exhaustion

Baseline’s LWC forecast is a simple linear projection. Enterprise runs ML-powered burn-rate prediction (linear regression or Prophet®) that accounts for seasonality, day-of-week patterns, and anomalies — so you see “you’ll hit Exceeded on the 23rd” four days in advance, not four hours.

● Multi-org — one budget, many orgs

Baseline tracks budget within a single Salesforce® org. Enterprise lets you pool and allocate a single flex-credit budget across multiple production orgs — sandbox, EMEA prod, APAC prod, acquired-company orgs, etc. — with org-level enforcement and a rolled-up executive dashboard. Essential for regulated multi-org deployments and acquisition-heavy organizations where unified AI governance is an audit requirement.

● AI-wide scope — beyond Agentforce™

Baseline governs the Agentforce™ runtime. Enterprise extends the same budget phase, circuit breaker, and model-tiering policies to every other AI surface that bills against your flex-credit wallet — Prompt Builder invocations from Flows, Einstein™ generative actions, custom LLM calls from Apex™, Marketing Cloud® AI, Data Cloud™ activations. One gateway, one policy, one budget for the whole organization’s AI footprint — applies to single-org and multi-org deployments alike.

● Observability — self-managed OpenTelemetry

Baseline uses native Salesforce® reports and a custom LWC. Enterprise emits standard OpenTelemetry traces, metrics, and logs from the gateway and lets you land them in whatever self-managed backend you already operate — so AI spend shows up on the same pane of glass your SRE / DevOps team already watches. Anomalies (a runaway subagent, a stuck loop, a sudden volume spike) page on-call instead of waiting for a weekly review.

Why OTel first: vendor-neutral wire format, no lock-in, trivial to re-point at a different backend later. Compatible backends (any of these work — this is not a recommendation, just the list of what the OTel wire format supports): self-hosted Grafana® / Prometheus®, SigNoz, Jaeger™, Tempo, Loki, Mimir, or any managed OTLP receiver such as Datadog®, New Relic®, Honeycomb®, Dynatrace®, Splunk® Observability Cloud, Elastic®, or Grafana Cloud™.

● Governor limits — offloaded

Baseline runs every consumption record through Apex + DML, which at >10K conversations/day starts bumping SOQL / DML governor limits. Enterprise moves that write-path off-platform entirely — Salesforce® fires a Platform Event, the gateway absorbs the write, and governor limits stop being a capacity ceiling.

When Enterprise pays for itself: The extra ~33 person-days and $200–$800/month cloud bill are justified when any of these apply — (1) conversation volume exceeds ~5K/day and latency matters, (2) you run multiple production orgs, (3) your ops team already runs a self-managed OpenTelemetry backend and wants AI spend on the same pane of glass, (4) manual weekly calibration is a recurring pain point, or (5) you have seen — or fear — a runaway-spend incident that a ~24-hour manual calibration window would have failed to catch.

Progressive Rollout Strategy

Production enforcement is turned on in four graduated stages rather than a single cutover. Each stage widens the blast radius while the team accumulates confidence that the shadow ledger matches reality, the phase engine transitions cleanly, and the circuit breaker trips only when it should.

Stage 1 · Shadow Mode
Gateway observes & logs only — no enforcement.
Duration: ~2 weeks
Blast radius: 0%
➤ Est. vs actual within ±25%
Stage 2 · Pilot
Enforcement on for one subagent or team.
Duration: ~2 weeks
Blast radius: 5–10%
➤ Pilot SLAs green, zero runaway spend
Stage 3 · Expansion
Enforcement rolls across all subagents & channels.
Duration: ~2 weeks
Blast radius: 50%
➤ No HITL capacity failures
Stage 4 · Full Enforcement
Production steady-state: all agents gated, all alerts live, weekly calibration.
Duration: Ongoing
Blast radius: 100%

B-2: Budget Phase Engine

The gateway’s decision brain. Reads the current ledger, computes the remaining-budget ratio, and maps it to one of four graduated phases using configurable thresholds from AI_Cost_Config__mdt. On transition, it publishes the AI_Budget_Phase_Change__e Platform Event (see A-2). Every other gateway component depends on this class for the current phase.

/**
 * BudgetPhaseEngine.cls
 * Computes budget phase from consumption ratio.
 * Fires AI_Budget_Phase_Change__e Platform Event on transitions.
 */
public with sharing class BudgetPhaseEngine {

    public enum Phase { ABUNDANT, CONSERVATIVE, EMERGENCY, EXCEEDED }

    public static Phase computePhase(String budgetPeriod) {
        AI_Budget_Ledger__c ledger = [SELECT Consumed_Estimate__c, Total_Ceiling__c
            FROM AI_Budget_Ledger__c
            WHERE Budget_Period__c = :budgetPeriod LIMIT 1];

        if (ledger == null || ledger.Total_Ceiling__c == 0) return Phase.ABUNDANT;

        Decimal ratio = 1 - (ledger.Consumed_Estimate__c / ledger.Total_Ceiling__c);
        AI_Cost_Config__mdt cfg = AI_Cost_Config__mdt.getInstance('Default');
        Decimal abundantThresh = cfg?.Abundant_Threshold__c != null ? cfg.Abundant_Threshold__c : 0.60;
        Decimal conservThresh = cfg?.Conservative_Threshold__c != null ? cfg.Conservative_Threshold__c : 0.20;

        if (ratio > abundantThresh)  return Phase.ABUNDANT;
        if (ratio > conservThresh)   return Phase.CONSERVATIVE;
        if (ratio > 0)               return Phase.EMERGENCY;
        return Phase.EXCEEDED;
    }

    public static void checkAndTransition(String budgetPeriod) {
        Phase current = computePhase(budgetPeriod);
        AI_Budget_Ledger__c ledger = [SELECT Current_Phase__c
            FROM AI_Budget_Ledger__c WHERE Budget_Period__c = :budgetPeriod LIMIT 1];

        String previousStr = ledger?.Current_Phase__c;
        if (previousStr == null) previousStr = 'ABUNDANT';

        if (current.name() != previousStr) {
            // Publish phase change event
            AI_Budget_Phase_Change__e evt = new AI_Budget_Phase_Change__e();
            evt.New_Phase__c = current.name();
            evt.Previous_Phase__c = previousStr;
            evt.Budget_Period__c = budgetPeriod;
            EventBus.publish(evt);

            // Update ledger
            ledger.Current_Phase__c = current.name();
            ledger.Last_Phase_Change__c = Datetime.now();
            update ledger;
        }
    }

B-3: Circuit Breaker Service

Implements the failsafe pattern described in the Circuit Breaker section. Maintains Closed/Open/Half-Open state in a Platform Cache partition with TTL-based auto-expiry. The Model Tier Router (B-4) checks isOpen() before every dispatch; if open, it returns DETERMINISTIC_FALLBACK immediately. Fails open on cache miss (returns Closed), ensuring cache eviction never blocks service.

/**
 * CircuitBreakerService.cls
 * Platform Cache-backed circuit breaker.
 * States: CLOSED (normal) → OPEN (tripped) → HALF_OPEN (testing)
 */
public with sharing class CircuitBreakerService {

    private static final String PARTITION = 'local.AIGateway';

    public static Boolean isOpen(String agentType) {
        String key = 'cb_' + agentType.replaceAll('[^a-zA-Z0-9]', '_');
        try {
            Integer failures = (Integer) Cache.Org.getPartition(PARTITION).get(key);
            if (failures == null) return false;
            AI_Cost_Config__mdt cfg = AI_Cost_Config__mdt.getInstance('Default');
            Integer threshold = cfg?.CB_Failure_Threshold__c != null
                ? (Integer) cfg.CB_Failure_Threshold__c : 3;
            return failures >= threshold;
        } catch (Exception e) {
            return false; // Fail-open on cache miss
        }
    }

    public static void recordFailure(String agentType) {
        String key = 'cb_' + agentType.replaceAll('[^a-zA-Z0-9]', '_');
        AI_Cost_Config__mdt cfg = AI_Cost_Config__mdt.getInstance('Default');
        Integer cooldown = cfg?.CB_Cooldown_Seconds__c != null
            ? (Integer) cfg.CB_Cooldown_Seconds__c : 300;
        try {
            Cache.OrgPartition part = Cache.Org.getPartition(PARTITION);
            Integer current = (Integer) part.get(key);
            part.put(key, (current != null ? current : 0) + 1, cooldown);
        } catch (Exception e) {
            // Log but do not block on cache errors
        }
    }

    public static void recordSuccess(String agentType) {
        String key = 'cb_' + agentType.replaceAll('[^a-zA-Z0-9]', '_');
        try {
            Cache.Org.getPartition(PARTITION).remove(key);
        } catch (Exception e) {
            // Fail silently
        }
    }
}

B-4: Model Tier Router

The gateway’s traffic cop. Takes the current budget phase, request complexity, and a prompt template prefix as inputs; returns the API name of the PromptTemplate tier to invoke. In Abundant phase, complex requests route to Tier 3; Conservative caps at Tier 2; Emergency permits only Tier 1; Exceeded returns DETERMINISTIC_FALLBACK. Also checks the Circuit Breaker — if open, the response is always deterministic regardless of phase. Exposed as @InvocableMethod for use in agent tools and Flows.

/**
 * ModelTierRouter.cls
 * Selects PromptTemplate tier based on budget phase + request complexity.
 * Exposed as @InvocableMethod for use in Agent Actions and Flows.
 */
public with sharing class ModelTierRouter {

    @InvocableMethod(label='Route to Model Tier' category='AI Cost Gateway')
    public static List<String> routeToTier(List<TierRequest> requests) {
        List<String> results = new List<String>();

        for (TierRequest req : requests) {
            // Check circuit breaker first
            if (CircuitBreakerService.isOpen(req.agentType)) {
                results.add('DETERMINISTIC_FALLBACK');
                continue;
            }

            String phase = req.budgetPhase != null ? req.budgetPhase : 'ABUNDANT';

            if (phase == 'EXCEEDED') {
                results.add('DETERMINISTIC_FALLBACK');
            } else if (phase == 'EMERGENCY') {
                results.add(req.templatePrefix + '_T1'); // Small model only
            } else if (phase == 'CONSERVATIVE') {
                // Medium for standard+complex; small for simple
                results.add(req.complexity == 'SIMPLE'
                    ? req.templatePrefix + '_T1'
                    : req.templatePrefix + '_T2');
            } else {
                // ABUNDANT: use appropriate tier for complexity
                if (req.complexity == 'COMPLEX') results.add(req.templatePrefix + '_T3');
                else if (req.complexity == 'STANDARD') results.add(req.templatePrefix + '_T2');
                else results.add(req.templatePrefix + '_T1');
            }
        }
        return results;
    }

    public class TierRequest {
        @InvocableVariable(required=true) public String agentType;
        @InvocableVariable(required=true) public String budgetPhase;
        @InvocableVariable(required=true) public String complexity; // SIMPLE, STANDARD, COMPLEX
        @InvocableVariable(required=true) public String templatePrefix; // e.g. "Case_Resolution"
    }
}

E-2: Shadow Ledger Database Schema

Mirrors the on-org custom objects but optimized for high-throughput writes and analytical queries. budget_ledger tracks one row per period with running estimates and phase. agent_usage logs per-conversation detail with indexed columns for the Grafana dashboards and burn-rate forecasting.

-- Shadow ledger and consumption tracking for the off-platform gateway

CREATE TABLE budget_ledger (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    budget_period   VARCHAR(20) NOT NULL UNIQUE,  -- e.g. '2026-Q2'
    total_ceiling   NUMERIC(18,2) NOT NULL,
    consumed_est    NUMERIC(18,2) NOT NULL DEFAULT 0,
    actual_consumed NUMERIC(18,2),
    calibration_fac NUMERIC(6,4) NOT NULL DEFAULT 1.0,
    current_phase   VARCHAR(20) NOT NULL DEFAULT 'ABUNDANT',
    last_phase_at   TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE agent_usage (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_type      VARCHAR(80) NOT NULL,
    conversation_id VARCHAR(80) NOT NULL,
    estimated_creds NUMERIC(10,2) NOT NULL,
    action_count    INT NOT NULL DEFAULT 0,
    turn_count      INT DEFAULT 0,

    budget_phase    VARCHAR(20),

E-4: Circuit Breaker Service (Redis)

Off-platform equivalent of the on-org Apex Circuit Breaker (B-3), backed by Redis for sub-millisecond reads and native TTL expiry. The INCR + EXPIRE pattern provides atomic failure counting with built-in cleanup. This makes Enterprise’s budget decisions sub-10ms versus Baseline’s 100–500ms.

// src/services/circuitBreakerService.js
import { redis } from '../shared/redis.js';

const DEFAULT_THRESHOLD = 3;
const DEFAULT_COOLDOWN  = 300; // seconds

export async function isOpen(agentType) {
    const key = `cb:${agentType}`;
    const failures = await redis.get(key);
    return failures !== null && parseInt(failures, 10) >= DEFAULT_THRESHOLD;
}

export async function recordFailure(agentType) {
    const key = `cb:${agentType}`;
    const count = await redis.incr(key);
    if (count === 1) {
        await redis.expire(key, DEFAULT_COOLDOWN);
    }
    return count;
}

export async function recordSuccess(agentType) {
    const key = `cb:${agentType}`;
    await redis.del(key);
}

Appendix F: Consumption Query Examples (Verified SOQL)

SOQL examples for the gateway’s primary consumption-data sources. Field names verified 2026-04-28 against the Salesforce® Help DLO reference articles ( TEUE, THEC, TET, TUTM, and the TenantUsageEntitlement Object Reference).

DLO field-name conventions. Digital Wallet DLO fields use lowercase with the __c custom-field suffix (e.g., unitsconsumed__c). Field names are case-insensitive in SOQL but the canonical casing matters for downstream tooling. Note one Salesforce® spelling quirk: eventime__c (not “eventtime”) — preserve this as-is; do not normalize.

DLO query gotchas. Free: querying via Data Explorer / Query Editor / custom reports built on TEUE. Cost-incurring: JDBC connector queries, joins with non-cost-excluded entities, using DLOs as data-transform inputs, querying custom DMOs based on Digital Wallet DLOs. All Digital Wallet metadata, connectors, and objects are read-only. Data Explorer stores data in UTC — the gateway’s reconciliation logic must normalize timestamps.

F-1: TenantEnrichedUsageEvent (TEUE) — Per-Event Resource-Level Consumption

The richest Digital Wallet surface. Use for per-feature / per-agent attribution and for consumption-tag-aware reporting. The gateway’s primary calibration source.

Source of truth for remaining credit balance and contract-pool drawdown order (expiring contracts drained first).

SELECT
    contractid__c,
    entitlementid__c,

F-6: Calibration Apex Class — Tying It Together

Skeleton Apex class showing how the gateway’s nightly auto-calibrator pulls from these sources to update the shadow ledger. Use as a structural starting point; field names and access patterns are verified, but Data Cloud SOQL invocation requires a Data Cloud query connection configured separately (see Data Cloud SOQL Query Guide).

// AutoCalibrator.cls — nightly job invoked by Schedulable trigger
// Pulls from TenantUsageEntitlement (Baseline) or TEUE/THEC/TUTM (Enterprise),
// updates AI_Cost_Config__mdt multipliers and AI_Budget_Ledger__c.Calibration_Factor__c.
public class AutoCalibrator implements Schedulable {

    public void execute(SchedulableContext sc) {
        // 1. Pull current entitlement balance (Baseline path — works without Data Cloud)
        List<TenantUsageEntitlement> ents = [
            SELECT Setting, MasterLabel, AmountUsed, CurrentAmountAllowed, EndDate
            FROM TenantUsageEntitlement
            WHERE EndDate >= TODAY
              AND (Setting LIKE '%FlexCredit%' OR Setting LIKE '%Agentforce%')
        ];

        // 2. (Enterprise tier only) Pull authoritative multipliers from TUTM via Data Cloud SOQL.
        //    Implementation note: Data Cloud SOQL is invoked via Connect API or REST,
        //    not via stock Database.query() — wire through the Pub/Sub gateway integration.
        Map<String, Decimal> multipliers = DataCloudClient.queryMultipliers();

        // 3. Compare shadow ledger to actuals
        AI_Budget_Ledger__c ledger = [
            SELECT Id, Consumed_Estimate__c, Calibration_Factor__c, Current_Phase__c
            FROM AI_Budget_Ledger__c
            WHERE Budget_Period__c = :BudgetPeriod.currentPeriodKey()
            LIMIT 1
        ];

        Decimal actualConsumed = 0;
        for (TenantUsageEntitlement e : ents) {
            actualConsumed += e.AmountUsed;
        }

        // 4. Update calibration factor (drift > ±25% triggers an alert per Risk R1)
        Decimal estimated = ledger.Consumed_Estimate__c;
        if (estimated != null && estimated > 0) {
            Decimal driftRatio = (actualConsumed - estimated).abs() / estimated;
            ledger.Calibration_Factor__c = (estimated == 0) ? 1 : actualConsumed / estimated;
            if (driftRatio > 0.25) {
                EventBus.publish(new AI_Calibration_Drift_Alert__e(
                    Drift_Ratio__c     = driftRatio,
                    Estimated__c       = estimated,
                    Actual__c          = actualConsumed,
                    Budget_Period__c   = ledger.Budget_Period__c
                ));
            }
        }
        update ledger;
    }
}
